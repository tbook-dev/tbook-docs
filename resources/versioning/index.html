Versioning

URL Prefix Versioning​

All API endpoints are versioned via a URL prefix:

https://api.tbook.com/v1/vaults
https://api.tbook.com/v2/vaults   (future)
  • v1 is the current stable version.
  • New major versions (v2, v3, ...) are introduced when breaking changes are required.
  • Multiple API versions may be active simultaneously during transition periods.

Compatibility Promise​

Within a Major Version (e.g. v1)​

Changes within a major version are additive only:

  • New endpoints may be added.
  • New optional fields may be added to request and response bodies.
  • New optional query parameters may be added.
  • New webhook event types may be introduced.
  • New enum values may be added to existing fields.

The following changes are never made within a major version:

  • Removing or renaming existing endpoints.
  • Removing or renaming existing response fields.
  • Changing the type or semantics of existing fields.
  • Making previously optional fields required.
  • Changing error codes for existing error conditions.

Across Major Versions (e.g. v1 to v2)​

Breaking changes are only introduced in new major versions. When a new major version is released, the previous version enters a deprecation period.

Deprecation Process​

When a new major API version is released:

  1. Announcement: A deprecation notice is published at least 6 months before the old version is retired.

  2. Sunset Header: All responses from the deprecated version include a Sunset HTTP header:

    Sunset: Sat, 01 Mar 2027 00:00:00 GMT

  3. Deprecation Header: Responses also include Deprecation: true.

  4. Migration Guide: A detailed migration guide is published alongside the new version.

  5. Retirement: After the 6-month notice period, the deprecated version returns 410 Gone.

SDK Versioning​

SDK packages follow Semantic Versioning 2.0.0:

MAJOR.MINOR.PATCH
  • MAJOR: Breaking changes to the public API surface.
  • MINOR: New features added in a backward-compatible manner.
  • PATCH: Bug fixes and performance improvements with no API changes.

SDK and API Version Mapping​

SDK VersionAPI VersionNotes
0.x.xv1Pre-release, API may shift
1.x.xv1Stable release
2.x.xv2Next major API version

Best Practices​

  1. Pin SDK versions using exact or tilde ranges (e.g. ~1.2.0) to avoid unexpected breaking changes.
  2. Monitor deprecation headers in API responses and plan migrations promptly.
  3. Subscribe to the changelog for advance notice of upcoming changes.
  4. Test against the latest SDK in CI to catch compatibility issues early.
