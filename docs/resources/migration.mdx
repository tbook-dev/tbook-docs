# Migration Guide

## Migration from v1 Single-Vault Routes

The old single-vault routes now return **301** (GET) or **307** (POST) redirects to the new multi-vault routes. Update your integration to avoid the redirect overhead.

| Old Route | New Route |
|-----------|-----------|
| `GET /v1/vault` | `GET /v1/vaults/:vaultId` |
| `GET /v1/vault/price` | `GET /v1/vaults/:vaultId/price` |
| `GET /v1/users/:pubkey` | `GET /v1/vaults/:vaultId/users/:pubkey` |
| `POST /v1/tx/deposit` | `POST /v1/vaults/:vaultId/tx/deposit` |
| `POST /v1/tx/redeem` | `POST /v1/vaults/:vaultId/tx/redeem` |
| `POST /v1/tx/claim` | `POST /v1/vaults/:vaultId/tx/claim` |
| `POST /v1/tx/cancel-deposit` | `POST /v1/vaults/:vaultId/tx/cancel-deposit` |

### Key Changes

- **vaultId in URL path**: The vault identifier is now part of the URL path instead of the request body.
- **TX endpoint request bodies**: No longer need the `vaultId` field — it is now part of the URL path.
- **User endpoints**: User account queries now include the vault ID in the path.

### Migration Steps

1. **Update API base paths**: Replace all old routes with the new multi-vault routes.
2. **Remove `vaultId` from request bodies**: For transaction endpoints, the `vaultId` is now in the URL path.
3. **Update user queries**: Change `/v1/users/:pubkey` to `/v1/vaults/:vaultId/users/:pubkey`.
4. **Test thoroughly**: Verify all endpoints work correctly with the new routes.

:::tip
The old routes will continue to work via redirects, but updating to the new routes eliminates the redirect overhead and ensures future compatibility.
:::
