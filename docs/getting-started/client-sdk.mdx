# Client SDKs

TypeScript SDKs for building and reading on each rail. They wrap the same REST contract — anything they do, you can also do with plain HTTP.

:::info SDK release status
The SDK packages ship together with the v1 sandbox release at version **0.2.0** (npm, `@tbook-dev` scope). Surfaces shown here are indicative until the packages are published; the [REST contract](/api-reference) is the normative interface.
:::

## Packages

| Package | Rail / purpose |
|---|---|
| `@tbook-dev/vault-sdk` | Solana rail: reads + unsigned transaction builders |
| `@tbook-dev/vault-sdk-sui` | Sui rail: reads + unsigned PTB builders (rcUSDP vault, XAUa AlphaVault) |
| `@tbook-dev/vault-react` | React hooks and drop-in components on top of the above |
| `@tbook-dev/vault-node` | Server-side: webhook verification + typed API client |

## What client SDKs are for

Use a client SDK when you build transactions **in the browser** against a connected wallet (wallet-adapter / dapp-kit), rather than through your backend:

```ts
// Indicative 0.2.0 surface — Sui rail
import { buildDeposit } from "@tbook-dev/vault-sdk-sui";

const { txBytesBase64, summary } = await buildDeposit({
  vault: "rcusdp-sui",          // from GET /v1/assets
  owner: wallet.address,
  amount: "100.00",
  network: "testnet",
});
// sign with @mysten/dapp-kit useSignAndExecuteTransaction
```

For server-driven integrations (most fintech apps, all omnibus/treasury setups), prefer the [REST API](/getting-started/rest-api) — it adds intent tracking, idempotency, and attribution automatically.
