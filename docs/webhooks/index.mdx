# Webhooks

Webhooks are the primary integration surface: every settlement, every per-account transaction milestone, and every vault state change is pushed to your endpoint. Build your UX on webhooks, not polling — settlement is asynchronous (T+1/T+3) by nature.

## Setup

### 1. Register an endpoint

```bash
curl -X POST $BASE/v1/webhooks \
  -H "Authorization: Bearer $TBK_SECRET_KEY" \
  -H "Content-Type: application/json" \
  -d '{"url": "https://your-server.example/tbook", "events": ["transaction.*", "vault.*"]}'
```

Save the returned `secret` (`whsec_…`) — it is shown once. Management endpoints: [Webhook Management](/api-reference/webhook-endpoints).

### 2. Verify every delivery {#verifying-signatures}

Each delivery carries:

```
X-TBook-Signature: t=1718438045,v1=5257a869e7ecebeda32affa62cdca3fa51cad7e77a0e56ff536d0ce8e108d8bd
```

`v1 = HMAC_SHA256(secret, "<t>.<rawBody>")`. Reject if the comparison fails (use a timing-safe compare) or if `|now − t|` exceeds 5 minutes.

**Node (`@tbook-dev/vault-node`):**

```ts
import { TBookVaultServer } from "@tbook-dev/vault-node";

const tbook = new TBookVaultServer({ webhookSecret: process.env.WHSEC });
app.post("/tbook", express.raw({ type: "*/*" }), (req, res) => {
  const event = tbook.webhooks.verify(req.body.toString(), req.header("X-TBook-Signature"));
  handle(event);          // do real work async
  res.sendStatus(200);    // ack fast
});
```

**Python (raw):**

```python
import hmac, hashlib, time

def verify(raw_body: bytes, header: str, secret: str) -> None:
    parts = dict(p.split("=", 1) for p in header.split(","))
    t, v1 = parts["t"], parts["v1"]
    if abs(time.time() - int(t)) > 300:
        raise ValueError("stale timestamp")
    expected = hmac.new(secret.encode(), f"{t}.".encode() + raw_body, hashlib.sha256).hexdigest()
    if not hmac.compare_digest(expected, v1):
        raise ValueError("bad signature")
```

Always verify against the **raw request body** — re-serializing JSON breaks the signature.

### 3. Acknowledge fast, process async

Return 2xx as soon as the signature checks out; do the real work on a queue. Non-2xx (or a timeout) triggers [retries](/webhooks/retry-policy).

## Payload envelope

```json
{
  "id": "evt_01h…",
  "type": "transaction.deposit.settled",
  "created": "2026-06-15T08:14:05Z",
  "data": { … }
}
```

- **Consume idempotently keyed on `id`** — the same event can be delivered more than once.
- Per-account events (`transaction.*`, `account.*`, `treasury.*`) are delivered **only to the organization that owns the account** and include your `externalId` so you can route without lookups.
- New event types appear over time — route unknown `type`s to a no-op handler ([forward compatibility](/getting-started/conventions#versioning--forward-compatibility)).

## Rotating secrets

`POST /v1/webhooks/:id/rotate-secret` issues a new secret; the old one stays valid for 24 hours in parallel so you can roll servers without dropping verification.
